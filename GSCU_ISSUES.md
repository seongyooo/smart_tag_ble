# GSCU 프로토콜 — 설계 고민 사항

> 현재 확정된 프로토콜(CLAUDE.md)에서 재검토가 필요한 부분들을 정리한 문서.

---

## 1. Tag Info (0x01) 충돌 문제

### 현재 상황

```
ESP32가 동시에 동작하는 것들:
├── GATT Server Advertising  ← 스마트폰이 태그를 찾을 수 있게
├── BLE Scanner              ← 0x02 브로드캐스트 수신
└── Tag Info (0x01) 방송     ← Passive ACK 역할
```

여러 태그가 동시에 0x01을 방송하면:

```
태그 1: 0x01 방송 중
태그 2: 0x01 방송 중
앱:     0x02 방송 중
→ 채널 혼잡 → 수신 신뢰성 저하
→ 가격 안 바뀌어도 계속 방송 → 전력 낭비
```

### 해결 방향 후보

```
A. 요청 기반:  평소 방송 안 함
               앱이 0x02 전송 후 → 태그에게 수신 요청
               태그가 그때만 0x01 응답

B. 시간 제한:  가격 수신 직후 3~5초만 0x01 방송
               이후 자동 중지

C. GATT Notify: 0x01 완전 제거
                Broadcast 수신 후 GATT 연결 → ACK Notify
                (충돌 없음, 단 속도 느림)
```

**미결**: 어떤 방식으로 ACK를 처리할지 확정 필요.

## 2. CRC 대신 Seq 기반 단순 ACK

### 현재 StateCRC 방식의 문제

```
StateCRC = CRC16(Price | Event | Start | End | Name)

문제점:
- 앱/태그 양쪽에서 동일한 CRC 계산 구현 필요 (복잡)
- CRC 불일치 시 어느 필드가 틀렸는지 모름
- 이름(NVS)까지 포함해야 해서 구현 난이도 높음
```

### Seq 방식 제안

```
앱이 패킷마다 Seq 번호 부여 (1바이트, 0~255 순환)
태그가 수신한 Seq 번호를 0x01로 그대로 돌려줌
앱이 자신이 보낸 Seq와 비교 → 일치하면 완료

동작 예시:
앱 → 0x02 (Seq=5, Tag03 1500원)
태그 → 0x01 (TagID=03, Seq=5)
앱 → Tag03 완료 처리 ✅
```

### 비교

```
StateCRC                    Seq
─────────────────────       ─────────────────────
전체 상태 동기화 확인 가능   마지막 수신 패킷만 확인
구현 복잡                   구현 단순
가격·이름·이벤트 통합 확인  필드별 확인 불가
```

**미결**: StateCRC 유지 vs Seq로 단순화 결정 필요.

## 3. Group ID 제거 및 물리적 구역 기반 업데이트

### 기존 Group ID의 역할

```
1. 패킷 필터링 → "내 그룹 패킷만 받겠다"
2. 구역 구분   → 앵커 태그와 연동
```

### 제거 방향

물리적 구역 내에서 Tag ID만으로 식별하면 Group ID 불필요.

```
기존: [FF FF 2B][Type 1B][GrpID 1B][Entry × 3]...  헤더 4B
변경: [FF FF 2B][Type 1B][Entry × 3]...             헤더 3B
→ 1바이트 추가 확보
```

### 확보된 1바이트 활용 옵션

```
A. Seq 번호    → ACK 단순화에 활용
B. HMAC 확장  → 3B → 4B (보안 강화)
C. Reserved   → 추후 확장 예약
```

### 구역 구분 대안

Group ID 제거 시 물리적 구역 구분은 **앱에서 수동 선택**으로 해결.

```
앱 화면에서 직원이 현재 구역 선택:
[과자 코너] [음료 코너] [생필품 코너]
     ↓
해당 구역 태그 목록만 Queue에 로드
broadcast는 전체로 뿌리되
앱 DB에서 해당 구역 태그만 관리
```

**미결**: Group ID 완전 제거 여부 확정 필요.

## 4. 앵커 태그 필요성 재검토

### 현재 설계

```
앵커 태그(0x03) → RSSI + 디바운스 → 구역 자동 감지
→ 해당 그룹만 broadcast
```

### Group ID 제거 시 앵커 태그도 불필요

```
Group ID 제거 → 그룹 필터링 없음
→ 앵커로 구역 감지해도 활용처 없음
→ 0x03 Type 자체가 불필요해질 수 있음
```

### 앵커 태그가 여전히 유용한 경우

```
완전 자동화 목표:
직원이 매대에 접근하면 자동으로 해당 구역 태그 Queue 로드
→ 이 경우 앵커 태그가 "구역 트리거" 역할
→ 단, Group ID 없이 앵커 Tag ID로 구역 매핑하는 방식 필요
```

**미결**: 앵커 태그 유지(자동화) vs 제거(수동 선택) 결정 필요.

## 5. 한 구역 내 다중 그룹 broadcast

### 현재 문제

```
현실 매장:
과자 코너 = 그룹A(새우과자류) + 그룹B(초콜릿류) 혼재
→ 그룹 필터링하면 한 번에 한 그룹만 업데이트 가능
→ 비효율적
```

### 해결 방향

```
그룹 개념 제거 → 구역 내 모든 태그를 Queue에 넣고 broadcast
태그는 자신의 Tag ID만 보고 반응
→ 자연스럽게 다중 그룹 동시 업데이트 가능
```

**Group ID 제거(§3)와 연계된 문제 → 함께 결정 필요.**

| 항목 | 현재 | 후보 |
|------|------|------|
| ACK 방식 | StateCRC (0x01) | A. 요청기반 / B. 시간제한 / C. GATT Notify / D. Seq |
| Group ID | 1바이트 유지 | 제거 → 1바이트 확보 |
| 앵커 태그 | 0x03 유지 | 제거(수동선택) vs 유지(자동화) |
| 구역 관리 | 그룹 기반 | 물리적 구역 + 앱 수동 선택 |
| 확보 바이트 활용 | - | Seq / HMAC 확장 / Reserved |
